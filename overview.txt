PROJECT HEALTH & STATUS ASSISTANT
Trello -> Dust sync: written overview

USE CASE
A Dust agent dedicated to PMs and eng leads that answers project health questions
directly, without opening Trello: which cards are overdue, which have no owner,
and a digest of each board's current state.

HOW THE SOLUTION WORKS
The sync script connects to the Trello API and pulls the boards, lists, and cards
(with due dates, members, labels, and comments) it needs. Each card is then
enriched with computed fields that don't exist in Trello's raw data: age, days
since last activity, whether it's overdue, and whether it's gone stale. The
enriched result is upserted into a Dust Data Source as one markdown document per
card, plus one digest document per board. A Dust agent is then pointed at that
Data Source and answers questions using this data directly.

HOW I VALIDATED AND TESTED IT
The enrichment logic (ages, overdue/stale flags) is covered by unit tests, since
it's the one part with real logic worth breaking. Everything else was validated
manually: a dry-run mode prints the generated documents without touching Dust, so
the Trello side could be checked first in isolation. Once the dry run looked
right, the script was run for real against Dust, the resulting documents were
checked in the Dust UI, and finally a Dust agent was created and fed with the
uploaded data to confirm it could answer real questions correctly.

ASSUMPTIONS, TRADEOFFS, LIMITATIONS
- Built for retrieval, not engagement: the agent reads project data, it doesn't
  act on Trello.
- This is a quick, purpose-built connector for this use case, not a generic
  template meant to support other project management tools.
- Every run resends all content rather than only what changed.
- No automation: the sync has to be run manually, there's no scheduling or
  webhook trigger.
- Not tested for scalability beyond a small demo board.
- Board digests are recreated daily rather than updated in place, so one new
  digest document accumulates per day.

WHY THIS APPROACH OVER THE ALTERNATIVES
This was chosen over building a Remote MCP Server for three reasons: it allows
the data to be enriched before the agent ever sees it, it's significantly less
expensive to build within the time available than standing up an MCP server, and
it's cheaper to run afterwards too, both to serve and in agent token cost.
